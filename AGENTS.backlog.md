# Blueprint Backlog

## Current Priorities
- [ ] Implement a non-destructive generated-consumer upgrade workflow with two canonical targets:
  - `make blueprint-upgrade-consumer`
  - `make blueprint-upgrade-consumer-validate`
  - Goal:
    - Upgrade existing generated-consumer repositories to latest blueprint-managed behavior (including runtime-identity/ESO/Keycloak changes) without losing consumer-owned modifications.
  - Required inputs and defaults:
    - `BLUEPRINT_UPGRADE_SOURCE` (default: current blueprint repository remote URL).
    - `BLUEPRINT_UPGRADE_REF` (required; tag/branch/commit pin for deterministic upgrades).
    - `BLUEPRINT_UPGRADE_APPLY` (default: `false`; dry-run planning by default, apply only when true).
    - `BLUEPRINT_UPGRADE_ALLOW_DIRTY` (default: `false`; refuse to run on dirty worktree unless explicitly allowed).
  - Ownership and safety contract:
    - Upgrade command must only touch blueprint-managed paths and init-managed/conditional scaffold paths allowed by `blueprint/contract.yaml`.
    - Upgrade command must never overwrite platform-owned consumer surfaces (`make/platform.mk`, `make/platform/*.mk`, `scripts/bin/platform/**`, `scripts/lib/platform/**`, `docs/platform/**`) via blind copy.
    - For files with local consumer edits in blueprint-managed areas, require 3-way merge behavior and emit conflict artifacts instead of destructive replacement.
    - Never delete consumer-created files by default; deletions require explicit opt-in flag and plan output.
    - Fail fast if conflict markers (`<<<<<<<`, `=======`, `>>>>>>>`) remain after apply.
  - Execution model:
    - Step 1: materialize pinned blueprint source in temp workspace and generate an upgrade plan.
    - Step 2: classify file actions (`create`, `update`, `merge-required`, `skip`, `conflict`) with ownership reason.
    - Step 3: apply planned actions when `BLUEPRINT_UPGRADE_APPLY=true`.
    - Step 4: run post-upgrade validation target automatically or as explicit follow-up.
  - Validation target contract (`blueprint-upgrade-consumer-validate`):
    - Must run at minimum:
      - `make quality-hooks-fast`
      - `make infra-validate`
      - `make quality-docs-check-core-targets-sync`
      - `make quality-docs-check-contract-metadata-sync`
      - `make quality-docs-check-runtime-identity-summary-sync`
      - `make quality-docs-check-module-contract-summaries-sync`
    - Must emit non-zero exit on any validation failure.
    - Must include a strict check that no unresolved merge markers exist.
  - Required artifacts (machine-readable + human summary):
    - `artifacts/blueprint/upgrade_plan.json`
    - `artifacts/blueprint/upgrade_apply.json`
    - `artifacts/blueprint/upgrade_validate.json`
    - `artifacts/blueprint/upgrade_summary.md`
  - Acceptance criteria:
    - Running upgrade in dry-run mode produces deterministic plan output and no file mutations.
    - Running upgrade in apply mode preserves consumer-owned modifications while applying blueprint-managed upgrades.
    - Validation target passes on a representative consumer repo with existing customizations.
    - Regression tests cover ownership boundaries, dirty-worktree guardrails, conflict handling, and artifact schema.
