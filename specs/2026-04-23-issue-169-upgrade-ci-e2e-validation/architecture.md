## Problem Statement
- What needs to change and why: The blueprint CI does not have a dedicated job for upgrade-flow validation. The `test_upgrade_fixture_matrix.py` runs inside `infra-contract-test-fast` (buried in the general `blueprint-quality` job), producing no separate artifact and no dedicated CI gate. Upgrade regressions are not surfaced as a distinct signal before release tag publication.
- Scope boundaries: New shell script + make target + CI job rendering + structural test. No changes to the existing upgrade Python scripts or test fixture matrix.
- Out of scope: Phase 2 extensions (#162, #163); live-tag upgrade validation; replacement of the existing `infra-contract-test-fast` invocation.

## Bounded Contexts and Responsibilities
- CI layer: `render_ci_workflow.py` owns the rendered `.github/workflows/ci.yml`. The new `upgrade-e2e-validation` job is a peer of `generated-consumer-smoke`, rendered by the same Python script and governed by the same `quality-ci-sync` / `quality-ci-check-sync` drift guard.
- Make layer: `quality-ci-upgrade-validate` target in `blueprint.generated.mk` (and its template source) is the local invocation surface; it delegates to `ci_upgrade_validate.sh`.
- Script layer: `ci_upgrade_validate.sh` owns the pytest invocation, artifacts directory creation, and JUnit XML path. It is a thin shell wrapper with no business logic.

## High-Level Component Design
- Domain layer: `test_upgrade_fixture_matrix.py` — existing test file; exercises resync + upgrade plan + upgrade apply via Python subprocess; not modified.
- Application layer: `ci_upgrade_validate.sh` — new shell script; creates the artifacts directory and invokes pytest with `--junitxml`; `quality-ci-upgrade-validate` make target — new entry in `blueprint.generated.mk.tmpl` / `blueprint.generated.mk`.
- Infrastructure adapters: `render_ci_workflow.py` — extended with `UPGRADE_E2E_VALIDATE_LANE` constant and `upgrade-e2e-validation` job rendering.
- Presentation/API/workflow boundaries: `.github/workflows/ci.yml` — rendered output; `upgrade-e2e-validation` job visible in GitHub Actions UI with JUnit artifact.

## Integration and Dependency Edges
- Upstream dependencies: `test_upgrade_fixture_matrix.py`, `scripts/lib/blueprint/upgrade_consumer.py`, `scripts/lib/blueprint/resync_consumer_seeds.py`, `tests/blueprint/fixtures/upgrade_matrix/`
- Downstream dependencies: `blueprint.generated.mk.tmpl` → `make/blueprint.generated.mk` (regenerated by `render_makefile.sh`); `render_ci_workflow.py` → `.github/workflows/ci.yml`
- Data/API/event contracts touched: `BLUEPRINT_CI_UPGRADE_ARTIFACTS_DIR` env var (new, consumed by `ci_upgrade_validate.sh` only)

## Non-Functional Architecture Notes
- Security: No external network calls in `ci_upgrade_validate.sh`; pytest invocations run entirely in local temp directories created by the test fixture setup.
- Observability: JUnit XML artifact uploaded to GitHub Actions as `upgrade-validate-junit` artifact; individual test names and pass/fail status visible in the Actions UI.
- Reliability and rollback: `set -euo pipefail` in `ci_upgrade_validate.sh`; failure propagates as non-zero exit. Rollback: remove the `upgrade-e2e-validation` job from `render_ci_workflow.py` and re-render.
- Monitoring/alerting: CI job failure on main push is the primary signal.

## Risks and Tradeoffs
- Risk 1: The `test_upgrade_fixture_matrix.py` test creates git repos in temp dirs; on a slow CI runner this can take 10-20 seconds. Acceptable for a push-only job.
- Tradeoff 1: Running `test_upgrade_fixture_matrix.py` twice on main pushes (once in blueprint-quality via hooks_fast, once in upgrade-e2e-validation) is accepted for explicit visibility and artifact upload. The duplication is intentional and bounded.
