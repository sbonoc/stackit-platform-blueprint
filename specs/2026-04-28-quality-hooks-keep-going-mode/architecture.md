## Problem Statement
- What needs to change and why: `scripts/bin/quality/hooks_fast.sh`, `hooks_strict.sh`, and `hooks_run.sh` together impose ~107 s per invocation on this repo and have four independent inefficiencies for the agent inner loop. (1) **Fail-fast multiplication:** with N independent failures, the gate runs N+1 times. (2) **Unconditional infra cost:** `infra-validate` (~9 s) and `infra-contract-test-fast` (~69 s) run on every invocation, regardless of whether the changeset can affect any infra contract — together ~73 % of total runtime. (3) **Phase-mismatch failure:** `quality-spec-pr-ready` runs as soon as the branch matches the SDD pattern, even at intake time when publish artifacts are by design scaffold; this fails-by-design and blocks any clean local run during Steps 1–6. (4) **Duplicate execution:** pre-commit local hooks already run `quality-docs-lint` (always) and `quality-test-pyramid` (on test-file changes); `hooks_fast.sh` then runs both again as separate `run_cmd` calls. The Step 5 implementation skill compounds these by leading agents to use `quality-hooks-fast` per slice when SDD only requires `test-unit-all` per slice.
- Scope boundaries: (a) a new shell helper `scripts/lib/shell/keep_going.sh` plus modifications to the three quality-hooks shell entry points; (b) two new shell helpers in the same file (or a sibling) for changed-path detection and SPEC_READY detection; (c) gating logic in `hooks_fast.sh` for `infra-validate`, `infra-contract-test-fast`, and `quality-spec-pr-ready`; (d) deletion of two redundant `run_cmd` lines in `hooks_fast.sh`; (e) make recipe doc-comment updates (no recipe code change required — env vars inherit); (f) edit to `.agents/skills/blueprint-sdd-step05-implement/SKILL.md` to direct per-slice gate; (g) ADR; (h) tests for helper, gating logic, and skill content.
- Out of scope: changing default fail-fast behavior of non-gated checks; aggregating pre-commit's internal hooks; parallel check execution; JSON output; new make targets; analogous gating for `hooks_strict.sh`; introducing a `quality-hooks-touch` lightweight target.

## Bounded Contexts and Responsibilities
- Context A (aggregation helper): `scripts/lib/shell/keep_going.sh` owns the per-check execution, output capture, status accumulation, and summary rendering. It is the single authority for what the keep-going summary block looks like and how per-check capture is handled. It exposes a small functional surface (`keep_going_init`, `run_check`, `keep_going_finalize`) and reads `QUALITY_HOOKS_KEEP_GOING` and `QUALITY_HOOKS_KEEP_GOING_TAIL_LINES` from the environment.
- Context B (hooks orchestration): `hooks_fast.sh`, `hooks_strict.sh`, `hooks_run.sh` own the ordered list of checks for their respective phases. They MUST keep the existing `run_cmd` invocations as the default code path for non-gated checks and dispatch through the helper only when keep-going is active. They own the pre-commit fail-fast invariant (only `hooks_fast.sh` runs pre-commit) and the cross-phase ordering (`hooks_run.sh` runs strict only after fast pre-commit succeeded).
- Context C (gating helpers): a new file `scripts/lib/shell/quality_gating.sh` (sibling to `keep_going.sh`) owns the changed-path computation (`quality_changed_paths` — emits the union of merge-base diff and working-tree diff), the path-gate predicate (`quality_paths_match_infra_gate <gating-set>` — returns 0 when the gating set is matched, 1 otherwise), and the spec-readiness predicate (`quality_spec_is_ready <spec-dir>` — returns 0 when `- SPEC_READY: true` is present in `<spec-dir>/spec.md`, 1 otherwise). The helpers are pure functions that read no global state beyond their arguments and the environment variable `QUALITY_HOOKS_FORCE_FULL`. `hooks_fast.sh` is the only consumer in v1; `hooks_strict.sh` is explicitly out of scope (see Architecture Out of Scope item).
- Context D (skill content): `.agents/skills/blueprint-sdd-step05-implement/SKILL.md` is the canonical source of the per-slice gate directive. The change in FR-014 is documentation-only and is owned by the SDD skill author; the implementation here updates the file content but does not change the skill's invocation contract, slash-command name, or runbook structure.

## High-Level Component Design
- Domain layer: `keep_going.sh` defines `KEEP_GOING_RESULTS` (parallel arrays for check name, status, duration, capture file path), `keep_going_active` (returns 0 when active), `run_check <name> -- <command...>` (executes, captures stdout+stderr to a temp file, records exit code and duration, re-emits the captured-tail to stderr on failure), and `keep_going_finalize` (prints the summary block, removes temp files via the EXIT trap, returns 0/1 by aggregate result). `quality_gating.sh` defines `quality_changed_paths`, `quality_paths_match_infra_gate <newline-list>`, and `quality_spec_is_ready <spec-dir>`. Both helpers are leaf modules — they depend on `logging.sh` and `exec.sh` only.
- Application layer: each hooks entry script after sourcing `keep_going.sh` calls `keep_going_init` near the start. For each check, the script either calls `run_cmd <command>` (default fail-fast — unchanged from today) or `run_check <name> -- <command>` (keep-going mode). A small dispatch wrapper at the top of each script picks one or the other based on `keep_going_active`. `hooks_fast.sh` adds gating dispatch around `infra-validate`, `infra-contract-test-fast`, and `quality-spec-pr-ready`: the gating predicate is consulted before either dispatch path is taken; if gated out and `QUALITY_HOOKS_FORCE_FULL` is not `true`, the script emits the skip log + metric and proceeds. The two redundant `run_cmd make ... quality-docs-lint` and `run_cmd make ... quality-test-pyramid` lines are deleted; their coverage continues via pre-commit local hooks. `hooks_run.sh` propagates the keep-going trigger and the `QUALITY_HOOKS_FORCE_FULL` value to its child invocations and orders pre-commit explicitly (fast first; strict only when pre-commit passed).
- Infrastructure adapters: per-check captures use `mktemp -t quality_hook_XXXX`; a single EXIT trap (registered via the existing `start_script_metric_trap` mechanism in `exec.sh`, extended to compose with the keep-going cleanup trap) removes the temp directory. Changed-path detection uses `git merge-base HEAD <main-branch>` (main branch resolved as the existing repo convention — `main`) and `git diff --name-only`; the result is cached in a script-local variable so multiple gates do not re-shell-out. SPEC_READY detection uses `pathlib`-style line scanning via `grep -E '^- SPEC_READY: true[[:space:]]*$' <spec-dir>/spec.md`. Branch detection (`git branch --show-current`) and `blueprint_repo_is_generated_consumer` continue to gate conditional checks; the gating decision happens before dispatch — gated-out checks are not added to the aggregator at all (so the summary reflects only checks actually executed).
- Presentation/API/workflow boundaries: the keep-going summary is plain text on stdout; the marker line is a stable contract (`===== quality-hooks keep-going summary =====`). Skip logs are plain text on stdout via `log_info` in the format `skipping <name>: <reason>`; skip metrics are emitted via the existing `log_metric` infrastructure. Exit codes are the only machine-readable signal consumed by `hooks_run.sh` and by `make`.

## Integration and Dependency Edges
- Upstream dependencies: `scripts/lib/shell/exec.sh` (`run_cmd`, `start_script_metric_trap`); `scripts/lib/shell/logging.sh` (`log_info`, `log_warn`, `log_error`, `log_metric`); `scripts/lib/blueprint/contract_runtime.sh` (`blueprint_repo_is_generated_consumer`).
- Downstream dependencies: `make/blueprint.generated.mk` recipes for `quality-hooks-fast`, `quality-hooks-strict`, `quality-hooks-run` and the corresponding template `scripts/templates/blueprint/bootstrap/make/blueprint.generated.mk.tmpl`. The recipes today are `@scripts/bin/quality/hooks_fast.sh` style with no env-var pass-through; make already exports the calling environment, so `QUALITY_HOOKS_KEEP_GOING=true make quality-hooks-fast` works without recipe changes — but recipe doc-comments need to mention it.
- Data/API/event contracts touched: none. The only externally observable contract is the summary marker line and the env var name.

## Non-Functional Architecture Notes
- Security: per-check captures live under `${TMPDIR:-/tmp}` with default user-only permissions; temp files are removed in the EXIT trap; no eval, no shell expansion of captured output (output is `printf '%s\n' "$captured"` only); no widening of script-level filesystem permissions.
- Observability: each check start logs via `log_info` (already present today via `run_cmd`'s `+ <command>` echo); on failure in keep-going mode, a 40-line tail (configurable) is re-emitted to stderr immediately. End-of-run metric `quality_hooks_keep_going_total` includes phase, status, and failed-check count.
- Reliability and rollback: default behavior is byte-identical when env var is unset and flag is not passed (FR-009). Rollback is to revert the patch — no state, no migration. The EXIT trap composes with the existing metric trap so cleanup runs even when checks are killed by signal.
- Monitoring/alerting: none — local developer + agent inner-loop tooling. No production-path SLOs.

## Risks and Tradeoffs
- Risk 1: The aggregation helper introduces a second control flow alongside the existing `run_cmd` calls. If the dispatch branch is buggy, the default path could regress. Mitigation: the default path remains a pure `run_cmd` call (the original line is preserved verbatim under an `if keep_going_active; then run_check ...; else run_cmd ...; fi` guard); a contract test asserts default invocations exit on first failure.
- Risk 2: Cascading false positives — a single root cause (e.g. a syntax error in a shared helper) can produce failures in every aggregated check that loads it. Mitigation: documentation makes the failure-ordering policy explicit ("fix the most fundamental failure first, re-run") and the summary block lists checks in execution order so the earliest failure is the first reported.
- Risk 3: Composing with `set -euo pipefail` — capturing the exit code of a failing command without aborting the script requires deliberate `||` patterns. Mitigation: the helper isolates the only `||` patterns to `run_check` itself; entry scripts continue to use the default `set -e` semantics for everything else.
- Risk 4 (path-gating false-skip): A change should have triggered `infra-validate` / `infra-contract-test-fast` but the path-matcher missed it (e.g. an indirect dependency through a non-listed path). Mitigation: the gating set is deliberately conservative (covers `infra/`, `blueprint/contract.yaml`, `scripts/lib/blueprint/`, `scripts/bin/blueprint/`, `scripts/templates/blueprint/`, `make/`, `apps/`, `pyproject.toml`, `requirements*.txt`); CI runs the full bundle unconditionally and is the safety net; `QUALITY_HOOKS_FORCE_FULL=true` provides the explicit override.
- Risk 5 (phase-gate timing): Gating `quality-spec-pr-ready` on `SPEC_READY: true` means an author who flips `SPEC_READY: true` prematurely (before publish artifacts are filled) gets a real failure rather than a friendly hint; agents and humans who never flip `SPEC_READY: true` get no signal until Step 7 explicitly invokes the check. Mitigation: this is correct by design — `SPEC_READY: true` is itself a sign-off-gated transition and Step 7 is the canonical caller; the readiness flag should only become true after publish artifacts are filled.
- Risk 6 (dedup regression): Removing `quality-docs-lint` / `quality-test-pyramid` from `hooks_fast.sh` makes those checks dependent on pre-commit being installed and configured. Mitigation: pre-commit is a documented prerequisite of the repo's quality model; the existing fallback branch in `hooks_fast.sh` is reframed (FR-013) to emit a `log_warn` so the user is told to install pre-commit; the gate does not silently succeed if pre-commit is missing.
- Risk 7 (skill-edit drift): Updating the Step 5 skill (FR-014) does not by itself prevent agents from running `quality-hooks-fast` per slice; a future skill edit could re-introduce the pattern. Mitigation: a contract test asserts the presence of the normative directive in `SKILL.md` so accidental removal is caught at gate time.
- Tradeoff 1: Two code paths (default fail-fast and keep-going) increase script complexity slightly but preserve the byte-identical default needed by CI. The alternative — replace `run_cmd` with `run_check` everywhere — would shift all of CI to keep-going semantics, which we explicitly do not want for production gates.
- Tradeoff 2: Path-gating in shell (vs in make) keeps the gating decision adjacent to the dispatch site (easy to reason about) at the cost of duplicating a small amount of logic that make's `target: $(deps)` dependency model could express. The alternative — make-based gating — was rejected in Option B for the same orchestration-cohesion reasons.
