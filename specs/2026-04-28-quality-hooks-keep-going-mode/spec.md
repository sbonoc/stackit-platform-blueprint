## Objective
- Business outcome: Reduce the agent inner-loop verification cost on `make quality-hooks-fast` / `make quality-hooks-run` along four independent axes: (1) eliminate the N+1 fail-fast re-run cost by adding an opt-in keep-going aggregation mode; (2) eliminate the dominant fixed cost (~78 of 107 measured seconds) on changes that cannot affect infra contracts by path-gating `infra-validate` and `infra-contract-test-fast`; (3) eliminate the false-positive Draft-PR failure of `quality-spec-pr-ready` at intake time by phase-gating it on `SPEC_READY: true`; (4) eliminate the duplicate execution of `quality-docs-lint` and `quality-test-pyramid` (already covered by pre-commit) inside `hooks_fast.sh`. The Step 5 implementation skill is updated to direct agents to the per-slice gate (`make test-unit-all`) rather than `quality-hooks-fast` per slice. The policy is propagated agent-agnostically via a single normative subsection in `AGENTS.md` (read by Claude, Codex, and any other agent that adheres to AGENTS.md), cross-linked by every skill that invokes `quality-hooks-*`, and reinforced by an opt-in env propagation kit (`.envrc` for direnv-using shells; `.claude/settings.json` env block for Claude Code; explicit AGENTS.md instruction for agents whose harness does not auto-load either) so that aggregation is on by default in any agent session.
- Success metric: With `--keep-going` (or `QUALITY_HOOKS_KEEP_GOING=true`), `quality-hooks-fast` and `quality-hooks-run` execute every independent downstream check even when earlier ones fail; the script exits non-zero with a summary section listing each failed check and its captured stderr tail. On a docs/spec-only commit, default `make quality-hooks-fast` runtime drops from the current ~107s baseline to under 15s on this repo (skipped infra checks logged with `status=skipped reason=no-relevant-paths`). On a Draft PR with `SPEC_READY: false`, `quality-spec-pr-ready` is skipped (logged with `status=skipped reason=spec-not-ready`) and the gate does not fail because of intentionally-scaffold publish artifacts. `quality-docs-lint` and `quality-test-pyramid` are no longer executed twice per `quality-hooks-fast` run. Default invocation (no flag, env var unset) preserves byte-identical fail-fast semantics for everything that still runs; pre-commit auto-mutating hooks remain fail-fast in all modes.

## Normative Requirements

### Functional Requirements (Normative)
- FR-001 `quality-hooks-fast` MUST accept a `--keep-going` CLI flag and MUST treat the environment variable `QUALITY_HOOKS_KEEP_GOING=true` (case-sensitive) as equivalent to passing the flag; any other value of the env var (including unset, empty, `false`, `0`) MUST behave as fail-fast (default).
- FR-002 `quality-hooks-strict` MUST accept the same `--keep-going` flag and `QUALITY_HOOKS_KEEP_GOING=true` env var with the same semantics as FR-001.
- FR-003 `quality-hooks-run` MUST accept the same `--keep-going` flag and `QUALITY_HOOKS_KEEP_GOING=true` env var, MUST forward the keep-going signal to both `hooks_fast.sh` and `hooks_strict.sh`, and MUST continue to invoke `hooks_strict.sh` after `hooks_fast.sh` returns non-zero EXACTLY WHEN keep-going is active.
- FR-004 In keep-going mode, `hooks_fast.sh` MUST run pre-commit (`pre-commit run --all-files`) as the first step in fail-fast semantics; if pre-commit reports failure (which on this codebase typically reflects file mutations performed by hooks), `hooks_fast.sh` MUST abort the keep-going run before any downstream check executes and MUST exit non-zero. Rationale: pre-commit MUST be assumed to rewrite files, and downstream checks observing the post-mutation tree would produce results that no longer correspond to the original input.
- FR-005 In keep-going mode, after pre-commit succeeds, `hooks_fast.sh` MUST execute every downstream independent check (shellcheck, `quality-root-dir-prelude-check`, `quality-infra-shell-source-graph-check`, `quality-sdd-check-all`, conditional `quality-spec-pr-ready`, `quality-docs-lint`, conditional `quality-ci-check-sync`, `quality-docs-check-changed`, `quality-test-pyramid`, `infra-validate`, `infra-contract-test-fast`) regardless of prior failures, capture each check's exit status and a tail of its captured output, and emit a consolidated summary block at the end.
- FR-006 In keep-going mode, `hooks_strict.sh` MUST execute every independent check (`infra-audit-version`, `apps-audit-versions`, conditional `blueprint-template-smoke`) regardless of prior failures and emit the same consolidated summary block format.
- FR-007 The consolidated summary block emitted by both scripts in keep-going mode MUST be prefixed with the literal marker line `===== quality-hooks keep-going summary =====`, MUST list each executed check with its status (`PASS` or `FAIL`) and runtime in seconds, and MUST be followed by EXACTLY ONE OF the trailer lines `===== all checks passed =====` (when zero checks failed) or `===== N check(s) failed =====` (when N > 0 checks failed).
- FR-008 In keep-going mode, scripts MUST exit `0` if and only if every executed check passed; otherwise they MUST exit `1`. Aggregating across `hooks_run.sh` (fast + strict), the combined exit MUST be `0` only when every check in both phases passed and pre-commit passed.
- FR-009 Default behavior (no `--keep-going` flag, env var unset or not equal to `true`) MUST be byte-identical to the pre-change scripts: each check runs via `run_cmd`, the first non-zero exit aborts the script via `set -e`, no summary block is printed, and exit codes match prior behavior. The `--keep-going` flag MUST be the only addition observable in default invocations.
- FR-010 The `--help` output of `hooks_fast.sh`, `hooks_strict.sh`, and `hooks_run.sh` MUST document the `--keep-going` flag, the `QUALITY_HOOKS_KEEP_GOING` env var, the `QUALITY_HOOKS_FORCE_FULL` env var, and the path-gating and phase-gating behaviors introduced by FR-011 / FR-012.
- FR-011 `hooks_fast.sh` MUST gate execution of `infra-validate` and `infra-contract-test-fast` on changed-path scope. The script MUST compute the changed path set as the union of (a) `git diff --name-only "$(git merge-base HEAD <main-branch>)" HEAD` and (b) `git diff --name-only` (working-tree changes vs HEAD). The two checks MUST run if and only if at least one changed path matches one of the gating prefixes/files (default set: `infra/`, `blueprint/contract.yaml`, `scripts/lib/blueprint/`, `scripts/bin/blueprint/`, `scripts/templates/blueprint/`, `make/`, `apps/`, `pyproject.toml`, `requirements*.txt`); otherwise both MUST be skipped and a `log_metric "quality_hooks_skip_total" "1" "phase=fast check=<name> reason=no-relevant-paths"` MUST be emitted. The env var `QUALITY_HOOKS_FORCE_FULL=true` MUST override skipping and force both checks to run regardless of changed paths. When git is unavailable or the merge-base resolution fails, the script MUST behave as if `QUALITY_HOOKS_FORCE_FULL=true` (fail-safe: run the checks).
- FR-012 `hooks_fast.sh` MUST invoke `quality-spec-pr-ready` ONLY when the resolved spec directory's `spec.md` contains the literal line `- SPEC_READY: true` (matched verbatim, leading `- ` and trailing whitespace tolerant). When the resolved spec directory does not exist, when the branch does not match the SDD pattern, or when `SPEC_READY` is anything other than `true` (including `false`, missing, or commented-out), the check MUST be skipped silently with a `log_metric "quality_hooks_skip_total" "1" "phase=fast check=quality-spec-pr-ready reason=<spec-not-ready|no-spec-dir|non-sdd-branch>"` log entry. The env var `QUALITY_HOOKS_FORCE_FULL=true` MUST override skipping and force `quality-spec-pr-ready` to run regardless of `SPEC_READY` state.
- FR-013 `hooks_fast.sh` MUST NOT explicitly invoke `quality-docs-lint` or `quality-test-pyramid` as separate `run_cmd` steps. Both checks remain covered by pre-commit local hooks declared in `.pre-commit-config.yaml` (`quality-docs-lint` always-run; `quality-test-pyramid` triggered on staged `tests/**.py`). When pre-commit is not installed (the existing fallback branch in `hooks_fast.sh`), the script MUST emit a `log_warn` directing the user to install pre-commit; the gate MUST NOT silently skip docs-lint and test-pyramid coverage in that case but MUST still execute the script's other downstream checks.
- FR-014 The implementation skill `.agents/skills/blueprint-sdd-step05-implement/SKILL.md` MUST be updated to: (a) explicitly direct that the per-slice gate is `make test-unit-all` (or the lane-specific runner derived from the spec's Implementation Stack Profile) and is NOT `make quality-hooks-fast`; (b) explicitly direct that `make quality-hooks-fast` is the slice-batch / pre-PR gate, run at the boundary between slices and again before publishing; (c) reframe the existing "Reproducible pre-commit failures" subsection so that `quality-hooks-fast` is referenced only in the pre-PR context, not as an inner-loop signal; (d) cross-link to the AGENTS.md canonical subsection introduced by FR-015 rather than restating policy.

The subsection MUST use the deterministic normative language policy (`MUST` / `MUST NOT` / `SHALL`). The subsection is the single canonical source for this policy; skills cross-link to it rather than restating.
- FR-016 The following six skill files MUST cross-link to the AGENTS.md `Quality Hooks — Inner-Loop and Pre-PR Usage` subsection rather than restating the policy: `.agents/skills/blueprint-sdd-step04-plan-slicer/SKILL.md`, `.agents/skills/blueprint-sdd-step05-implement/SKILL.md` (in addition to FR-014), `.agents/skills/blueprint-sdd-step07-pr-packager/SKILL.md`, `.agents/skills/blueprint-consumer-upgrade/SKILL.md`, `.agents/skills/blueprint-consumer-ops/SKILL.md`, and the matching reference checklists under `references/` for skills that have them. ### Non-Functional Requirements (Normative)
- NFR-SEC-001 The keep-going implementation MUST NOT introduce shell injection vectors via the captured per-check output (output is captured to a file, then printed verbatim; no eval, no expansion). The implementation MUST NOT widen filesystem permissions or write outside `${TMPDIR:-/tmp}` for transient capture files; capture files MUST be removed in an `EXIT` trap.
- NFR-OBS-001 In keep-going mode, each check's start MUST be logged via `log_info` with the check name; on failure, the captured stderr tail (last 40 lines, configurable via `QUALITY_HOOKS_KEEP_GOING_TAIL_LINES` integer env var, default `40`) MUST be re-emitted to the script's stderr immediately after the check completes so the failure is visible in scrollback as well as in the final summary. A `log_metric "quality_hooks_keep_going_total" "1" "status=success|failure phase=fast|strict failed_checks=<count>"` MUST be emitted at the end of each script run in keep-going mode.
- NFR-REL-001 If a check process is killed by signal (e.g. `SIGTERM`, `SIGINT`) during keep-going execution, the script MUST treat the surrounding wait as terminal: the EXIT trap MUST run and the script MUST exit non-zero; in-flight per-check capture files MUST be cleaned up. Subsequent checks MUST NOT execute after a fatal signal.
- NFR-OPS-001 `--keep-going` MUST be discoverable via `--help` for all three scripts (FR-010); the make target `help` output (existing `## Run …` doc-comments in `make/blueprint.generated.mk`) MUST be amended to mention the env var on `quality-hooks-fast`, `quality-hooks-strict`, and `quality-hooks-run` so `make help` lists it.
- NFR-OBS-002 Path-gated and phase-gated skips MUST emit a structured `quality_hooks_skip_total` metric with labels `phase`, `check`, and `reason` so that "what was skipped and why" is auditable in any run; skipped checks MUST also emit a single `log_info` line in the format `skipping <check>: <reason>` so the skip is visible in scrollback.
- NFR-OPS-002 `QUALITY_HOOKS_FORCE_FULL=true` MUST be discoverable via `--help` on all three scripts and via the `make help` doc-comment on `quality-hooks-fast`, `quality-hooks-strict`, `quality-hooks-run`; the operations doc MUST document it alongside `QUALITY_HOOKS_KEEP_GOING`.

## Normative Option Decision
- Option A: Implement keep-going as a shell helper (`run_check_aggregating`) in a new file `scripts/lib/shell/keep_going.sh` sourced by `hooks_fast.sh`, `hooks_strict.sh`, and `hooks_run.sh`. Each script declares its check list as `name|command` pairs and dispatches via the helper, which captures stdout+stderr to a per-check temp file, records exit status and duration, and at the end prints the summary block. Default mode keeps the existing `run_cmd <thing>` invocations unchanged. Path-gating (FR-011) and phase-gating (FR-012) decisions happen before dispatch; gated-out checks are not invoked at all and not added to the keep-going aggregator. Dedup (FR-013) deletes the redundant `run_cmd` lines outright. Skill clarification (FR-014) is a documentation-only edit to `.agents/skills/blueprint-sdd-step05-implement/SKILL.md`.
- Option B: Extract each check into its own make target and use `make -k` (keep-going) at the top level. Rely on make's existing aggregation semantics. Implement gating via make conditionals.
- Selected option: OPTION_A
- Rationale: Option B requires re-architecting all current shell-driven check invocations into make targets and depends on make's keep-going behavior, which aggregates exit codes but does not produce a structured per-check summary, does not capture per-check output for tail-replay, does not handle the conditional checks (branch-pattern-gated `quality-spec-pr-ready`, repo-mode-gated `quality-ci-check-sync` and `blueprint-template-smoke`, the new path-gating and phase-gating from FR-011 / FR-012), and does not enforce the pre-commit-first-and-fail-fast invariant from FR-004. Option A keeps the existing single-responsibility shell entry points, isolates aggregation logic in one helper, expresses gating as small shell functions adjacent to the dispatch site, and preserves byte-identical default behavior for non-gated checks.

## Informative Notes (Non-Normative)
- Context: Measured baseline on this repo (`time make quality-hooks-fast` on a docs/spec-only working tree): ~107 s total. Decomposition: pre-commit ~3.5 s, shellcheck ~5–10 s, every other quality-* check <1 s, `infra-validate` ~9 s, `infra-contract-test-fast` ~69 s. The two infra checks account for ~78 s (~73%) of the runtime and are not affected by docs/spec/agent-skill changes. A typical agent inner loop on a mid-size SDD slice incurs this cost N+1 times because of fail-fast aggregation. Combined with the dedup of two duplicate sub-checks and the phase-gating of `quality-spec-pr-ready` (which currently fails-by-design on Draft PRs at intake), the four FRs together address the dominant axes of inner-loop verification cost.
- Tradeoffs: (a) Aggregated reports (FR-005..FR-008) can include cascading false positives when a single root cause produces multiple downstream failures. The recommended workflow (operations doc + ADR) is to fix the most fundamental failure first and re-run, rather than mass-applying fixes. Pre-commit fail-fast (FR-004) prevents the worst case. (b) Path-gating (FR-011) trades a small risk of false-skips (a change that ought to have triggered the infra checks but did not match the gating set) against a large speedup on the common case; the mitigation is a precise, deliberately conservative gating set, the `QUALITY_HOOKS_FORCE_FULL=true` override, and CI which always runs the full bundle and is the safety net. (c) Phase-gating `quality-spec-pr-ready` (FR-012) trades the early signal of a malformed publish artifact for the elimination of a false-positive at intake; the mitigation is that `quality-spec-pr-ready` becomes mandatory once `SPEC_READY: true` and Step 7 runs it explicitly. (d) Dedup (FR-013) trades a near-zero risk that someone removes pre-commit's docs-lint or test-pyramid hook for a small runtime saving and reduced log noise; mitigation is the FR-013 fallback that emits a `log_warn` when pre-commit is not installed.
- Clarifications: none

## Explicit Exclusions
- Excluded item 1: Changing the default behavior of `quality-hooks-fast` / `quality-hooks-strict` / `quality-hooks-run` for non-gated checks is out of scope. CI and pre-commit invocations remain fail-fast for the steps that still execute. The new gating in FR-011 / FR-012 is the only behavior change in default mode.
- Excluded item 2: Aggregating pre-commit's individual hook failures inside pre-commit itself is out of scope. Pre-commit already has its own internal aggregation (`pre-commit run --all-files` reports all hooks); we treat its overall exit code as a single fail-fast signal because it can mutate files.
- Excluded item 3: Parallel execution of independent checks is out of scope. Aggregation is sequential. Parallelism is a separate optimization that can build on this foundation later.
- Excluded item 4: Building a structured (JSON) machine-readable summary for the agent harness is out of scope for v1. The plain-text summary block is the v1 contract; a JSON variant can be added later if agent harness integration requires it.
- Excluded item 5: Introducing a separate `quality-hooks-touch` (lightweight inner-loop) target is out of scope. With FR-011 path-gating, FR-013 dedup, and FR-014 skill clarification, default `make quality-hooks-fast` on docs/spec-only commits drops to under 15 s — a separate target adds maintenance surface for marginal gain. May be revisited if measured runtime is still insufficient after this work item ships.
- Excluded item 6: Removing `bash -n` from pre-commit (since `shellcheck --severity=error` in `hooks_fast.sh` covers it) is out of scope. `bash -n` is near-instant and provides per-file feedback during the pre-commit auto-fix pass; removing it would not measurably speed anything up.
- Excluded item 7: Modifying `hooks_strict.sh` for path-gating (analogous to FR-011) is out of scope. The strict checks (`infra-audit-version`, `apps-audit-versions`, `blueprint-template-smoke`) are run pre-push and pre-PR, not in the agent inner loop, so the cost-vs-correctness tradeoff is different. May be revisited as a separate work item if pre-push wait time becomes a problem.
