# Hardening Review

## Repository-Wide Findings Fixed
- Finding 1: Hardcoded baseline component IDs (`backend-api`, `touchpoints-web`) and manifest filenames in `scripts/lib/platform/apps/catalog_scaffold_renderer.py` and the catalog manifest template prevented consumers from declaring their own apps without forking the renderer. Fixed by deriving `deliveryTopology.workloads` and `runtimeDeliveryContract.gitopsWorkloads` from `apps/descriptor.yaml` via the new `render_delivery_workloads_block` / `render_gitops_workloads_block` helpers and the repeatable `--component-image` / `--component-image-env-var` CLI flags. The renderer source no longer contains baseline-id literals; `tests/blueprint/test_app_catalog_descriptor_rendering.py::test_renderer_module_has_no_hardcoded_baseline_ids` enforces this.
- Finding 2: The blueprint upgrade pipeline lacked an explicit consumer-owned app metadata source, leaving prune classification dependent on path-prefix heuristics (`_is_consumer_owned_workload`) and kustomization-ref scanning (`_is_kustomization_referenced`). Fixed by introducing `apps/descriptor.yaml` as the canonical app declaration contract and a new `consumer-app-descriptor` prune-guard branch ahead of the deprecated bridge guards. Existing guards remain as fallback for the two-minor-release migration window.
- Finding 3: Existing generated consumers had no migration aid for adopting `apps/descriptor.yaml`. Fixed by `generate_suggested_descriptor` + `write_suggested_descriptor_artifact`, which emit `artifacts/blueprint/app_descriptor.suggested.yaml` from the consumer's existing `apps/kustomization.yaml` resources without ever writing to the consumer working tree.

## Observability and Diagnostics Changes
- Metrics/logging/tracing updates: apply artifact JSON gains a `consumer_app_descriptor_count` summary counter; descriptor validation produces deterministic stderr/error-list messages naming the descriptor app, component ID, manifest kind, and offending path/value; existing `consumer_kustomization_ref_count`, `tf_dedup_count`, and `deduplication_log` counters from PR #212 remain unchanged.
- Operational diagnostics updates: when `apps/descriptor.yaml` is absent during upgrade apply, `apply_payload.suggested_app_descriptor_artifact` records the relative path of the emitted advisory artifact; the `_is_consumer_owned_workload` docstring and the rendered `apps/catalog/manifest.yaml` both lead with `DEPRECATED` markers naming the migration window and the `after: consumer-app-descriptor-adoption` removal trigger.

## Architecture and Code Quality Compliance
- SOLID / Clean Architecture / Clean Code / DDD checks: descriptor loader (`scripts/lib/blueprint/app_descriptor.py`) is a pure parsing/validation module with no I/O beyond the descriptor file itself; verification helpers (`verify_resolved_manifests_exist`, `verify_kustomization_membership`) take callable dependencies (`kustomization_resources`) by injection so the contract validator wiring stays decoupled from filesystem helpers. Catalog renderer logic iterates descriptor records without baseline-id knowledge; image binding is supplied by the caller via repeatable CLI flags. Upgrade guard ordering is explicit and documented in the classifier comment block (descriptor → consumer-owned-workload → kustomization-ref → delete).
- Test-automation and pyramid checks: 53 new unit/integration tests across slices 2–5 (`test_app_descriptor_loader.py`, `test_app_catalog_descriptor_rendering.py`, `test_consumer_app_descriptor_upgrade.py`, `test_consumer_app_descriptor_deprecations.py`); all sit at the lowest valid layer with synthetic temp directories, no live cluster or process boundaries. Test pyramid contract updated to register the four new test files. Pyramid ratios remain healthy (unit 93.89%, integration 4.78%, e2e 1.33%).
- Documentation/diagram/CI/skill consistency checks: ADR Mermaid `flowchart TD` (already in PR description) reflects the descriptor flow; consumer docs (`app_onboarding.md`, `quickstart.md`, `troubleshooting.md`) gain a descriptor reference, schema table, validation/error-pattern guidance, and the suggested-artifact migration workflow; blueprint docs (`execution_model.md`) extend the prune-guard description from three to four layers and document the suggested-artifact behavior. Bootstrap template mirrors are in sync (`make quality-docs-check-changed` clean).
